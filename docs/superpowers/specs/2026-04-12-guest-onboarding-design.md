# Guest Onboarding — Empty Wardrobe Redesign

**Date:** 2026-04-12  
**Status:** Approved  
**Scope:** `src/components/WardrobeTab.jsx`, new `src/components/GuestLanding.jsx`, new `src/utils/demoData.js`

---

## Problem

When a guest visits Vesti with an empty wardrobe, they see:
- "0 Pieces. (0/10)" — looks broken, not inviting
- Filters and search bar with nothing to filter
- No explanation of what the app does
- No clear first action

Goal: replace the blank state with a guided experience that drives a first action in under 5 seconds, without requiring login.

---

## Approved Design

**Approach B (demo preview first) + Option C (Try demo → loads real data, navigates to Daily tab)**

## Section 2: Demo Mode — Header Badge

When demo mode is active, a small amber **"DEMO"** pill appears inline next to the Vesti wordmark in `App.jsx`'s header. Implemented in `AuthenticatedApp` which already reads `isGuest` from `useTier`.

```jsx
{isDemoMode && (
  <span
    title="You're in demo mode"
    onClick={clearDemo}
    style={{
      fontSize: 8, fontWeight: 700, letterSpacing: 0.8,
      color: "#FBBF24",
      background: "rgba(251,191,36,0.12)",
      border: "1px solid rgba(251,191,36,0.25)",
      borderRadius: 20, padding: "2px 7px",
      cursor: "pointer",
    }}
  >
    DEMO
  </span>
)}
```

Clicking the badge calls `clearDemo()` — which removes all demo data and navigates back to Wardrobe tab.

`isDemoMode` is a boolean stored in localStorage: `vesti_guest_<guestId>_demo_mode = "1"`. It is read and exposed via a new `useDemoMode()` hook (or inline in `useAuth` context — TBD during implementation).

## Section 3: Demo Data + "Try Demo" Action

### Demo wardrobe (`src/utils/demoData.js`)

12 pre-built items, fully valid against the wardrobe schema:

| Layer | Items |
|---|---|
| Base | White linen shirt, Black merino tee, Cream knit |
| Mid | Grey cashmere roll-neck |
| Outer | Black bomber, Navy trench coat |
| Bottom | Navy slim trousers, Khaki chinos, Black jeans |
| Footwear | White leather sneakers, Brown leather boots, Black dress shoes |

Each item has: `id`, `n`, `c`, `col`, `b`, `l`, `occ`, `w`, `t: "Yes"`, `img: ""`, `_demo: true`

The `_demo: true` flag allows easy bulk-removal.

### Demo trip

Use the existing `SEED_TRIP` from `src/data/trip.js` (Australia & NZ 2026). No new data needed.

### Demo outfits (pre-assigned, 3 days)

Stored in `src/utils/demoData.js` as a static object keyed `{ [tripId]: { [dayId]: { daytime, evening } } }`. Uses the demo item IDs. Days d01–d03 are pre-assigned; d04+ remain empty (to be generated by user).

### `loadDemo(guestId, setTab)` function

```
1. Write demo items to localStorage: vesti_guest_<guestId>_wardrobe
2. Write demo trip to localStorage: vesti_guest_<guestId>_trips_local
3. Write demo outfits to localStorage: vesti_guest_<guestId>_outfits_<tripId>
4. Set demo flag: vesti_guest_<guestId>_demo_mode = "1"
5. Call setTab("daily")  ← navigates to Daily tab
6. Dispatch CustomEvent "vesti-demo-loaded" so hooks re-read localStorage
```

All writes happen synchronously before `setTab` so the Daily tab sees data immediately.

### `clearDemo(guestId)` function

```
1. Remove vesti_guest_<guestId>_wardrobe
2. Remove vesti_guest_<guestId>_trips_local
3. Remove vesti_guest_<guestId>_outfits_v1_<DEMO_TRIP_ID>
   (exact key format matches useOutfits lsKey for guests)
4. Remove vesti_guest_<guestId>_demo_mode
5. Dispatch "vesti-demo-cleared"
6. Navigate to wardrobe tab
```

`DEMO_TRIP_ID = "trip_aus_nz_2026"` — same constant used in `useTripStore.js`.

## Section 4: Save CTA in Daily Tab (Behavior-Triggered)

### Placement

Below the last outfit slot card in the Daily tab content area. Rendered only when `isGuest && isDemoMode && saveCtaVisible`.

### Card design

```
┌─────────────────────────────────────┐
│  Your trip is ready to save         │  (13px, 700)
│  Sign in to keep it across devices  │  (12px, T.mid)
│                                     │
│  [ G  Continue with Google ]        │  (white btn, Google icon)
│                                     │
│  Free forever · No credit card      │  (10px, T.light, centered)
└─────────────────────────────────────┘
border: 1px solid rgba(10,132,255,0.22)
background: T.surface
border-radius: 10px
```

**Copy rationale:**
- "Your trip is ready to save" — ownership framing, feels like they built something
- "Sign in to keep it across devices" — practical benefit, not sales language
- "Free forever · No credit card" — removes friction and fear before the tap

"Continue with Google" calls the existing Supabase Google OAuth sign-in. On successful login, `migrateGuestData` runs automatically (already implemented in AuthContext) — demo data migrates with it.

No email/OTP in this card — keep it to one action. The existing SIGN IN button in the header handles other auth methods.

### Micro animation

When `saveCtaVisible` flips from `false` to `true`, the card animates in:

```css
@keyframes slideUpFade {
  from { opacity: 0; transform: translateY(14px); }
  to   { opacity: 1; transform: translateY(0); }
}
/* Applied to the card wrapper */
animation: slideUpFade 0.35s cubic-bezier(0.16, 1, 0.3, 1) forwards;
```

This keyframe is added inline (via a `<style>` tag or the existing global `<style>` block in `App.jsx`). No external animation library needed.

### Behavior trigger

`saveCtaVisible` starts as `false`. It becomes `true` (permanently for the session) when **any one** of these fires:

1. **Regen click** — user clicks ↺ on any outfit slot
2. **Day navigation** — user browses to Day 2 or beyond (day index ≥ 1)
3. **Item click** — user taps any outfit swatch/item card in demo mode
4. **Time in demo** — 20–25 seconds elapsed since `loadDemo()` ran (use 22s)

Trigger state lives in component state (not localStorage) — resets if user clears demo and re-enters. Timer is a `useEffect` with `setTimeout(fn, 22000)`.

## Section 6: WardrobeTab Changes

**Render strategy:** Do NOT early-return from WardrobeTab. The add-item modal (`adding` state) lives at the bottom of the component and must always be reachable. Instead, swap the hero section conditionally:

```jsx
// Inside the return, replace the hero block:
{isGuest && wardrobe.length === 0 && !isDemoMode ? (
  <GuestLanding
    onAddItem={() => setAdding(true)}
    onTryDemo={onTryDemo}   // ← callback prop from App.jsx
  />
) : (
  // existing hero + search + filters + grid
  <div>…</div>
)}

{/* Add-item modal always present regardless of landing/demo state */}
{adding && <AddItemForm … />}
```

`onTryDemo` prop threads from `App.jsx` → `WardrobeTab` → `GuestLanding`. In `App.jsx`, `onTryDemo` calls `loadDemo(guestId)` then `setTab("daily")`.

When in demo mode (`isDemoMode = true`), the wardrobe renders normally — demo items show as real items. No special UI treatment needed.
